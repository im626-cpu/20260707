# 캠퍼스 배달 메이트 매칭 서비스 — PRD (v0.1, MVP)

## 1. 개요

같은 대학 캠퍼스 내에서 배달 음식을 혼자 시키기엔 최소 주문 금액이 부담스러운 학생들이,
"어디서 언제 무엇을 시킬지"를 게시판에 올리고 다른 학생들과 함께 주문할 사람을 모아
최소 주문 금액을 채우고 배달비를 아낄 수 있게 해주는 매칭 서비스.

### 배경 / 문제 정의
- 기숙사·자취방 등 배달 최소 주문 금액을 혼자서는 채우기 어려운 경우가 많음.
- 같은 건물/동에 함께 시킬 사람이 있어도 찾을 방법이 마땅치 않음 (에브리타임 자유게시판 등 비정형적 방식에 의존).
- 위치·시간·메뉴 세 조건이 동시에 맞아야 하므로 전용 매칭 도구가 필요함.

### 목표
- 게시글 작성 → 참여 신청 → 방장 승인 → 최소 주문 금액 도달 시 매칭 완료 → 그룹 채팅으로 이어지는 흐름을 안정적으로 제공.
- 실제 주문/결제는 앱 밖(배달앱)에서 대표 1인이 진행하는 것을 전제로, "사람을 모으는 것"에만 집중.

### 성공 지표 (MVP 기준, 참고용)
- 게시글당 매칭 완료율 (최소 주문 금액 도달 후 마감된 비율)
- 게시글 작성 후 매칭 완료까지 걸리는 평균 시간
- 재방문/재작성 사용자 비율

---

## 2. 타겟 사용자 및 범위

- **대상**: 단일 대학 캠퍼스 재학생 (학교 이메일 인증 보유자)
- **범위**: 초기에는 학교 1곳만 지원. 단, 추후 여러 학교로 확장 가능하도록 데이터 모델에 `school_id` 개념을 처음부터 포함.
- **인증**: 학교 이메일(`*.ac.kr` 등) 인증 후 가입. 인증되지 않은 사용자는 게시글 열람만 가능하고 작성/참여는 불가 (Out of scope 세부 정책은 §8 참고).

---

## 3. 용어 정의

| 용어 | 정의 |
|---|---|
| 모임(게시글) | 한 사용자가 "이 가게에서 이 메뉴를 이 시간에 시키고 싶다"고 올리는 게시물 |
| 방장 | 모임을 작성한 사용자. 참여 신청을 승인/거절할 권한을 가짐 |
| 참여자 | 방장의 승인을 받아 모임에 합류한 사용자 |
| 매칭 완료 | 승인된 참여자들의 예상 주문 금액 합산이 모임에 설정된 최소 주문 금액 이상이 된 상태 |
| 정산 | 실제 주문 후 인원별 금액을 나누는 절차 (MVP 범위 아님, §8 참고) |

## 4. 핵심 사용자 흐름

### 4-1. 모임 만들기 (방장)
1. 로그인 후 "모임 만들기" 클릭
2. 위치(건물 드롭다운 + 상세 위치 자유 입력), 식사 가능 시간대(시작~종료, 예: 12:00~12:30), 가게명, 메뉴 설명, 배달료, 최소 주문 금액을 입력
3. 본인이 시킬 메뉴의 가격(예상 주문 금액)을 함께 입력 (본인도 참여자 1인으로 카운트)
4. 게시글 등록 → 게시판에 "모집중" 상태로 노출

### 4-2. 참여하기 (참여자)
1. 게시판에서 위치/시간/메뉴로 게시글 탐색·필터링
2. 원하는 모임 상세 진입 → 예상 주문 금액 입력 후 참여 신청
3. 방장의 승인/거절 대기
4. 승인되면 참여자 목록에 추가되고, 누적 금액이 갱신됨

### 4-3. 매칭 완료
1. 승인된 참여자들의 예상 주문 금액 합산이 최소 주문 금액 이상이 되면 게시글 상태가 자동으로 "매칭 완료"로 전환
2. 매칭 완료 시 게시판 목록에서 자동 마감(더 이상 신규 신청 불가, 목록에서 숨김 또는 "마감" 표시)
3. 이 시점에 아직 방장이 승인하지 않은 대기(pending) 신청이 남아있다면, 시스템이 해당 신청 전건을 자동으로 거절 처리
4. 매칭 완료와 동시에 참여자 전원이 포함된 그룹 채팅방이 자동 생성됨
5. 그룹 채팅방에서 실제 메뉴 확정, 계좌번호 공유 등은 사용자들이 자율적으로 진행 (앱은 채팅만 제공)

### 4-4. 마감/취소
- 식사 가능 시간이 지나면 매칭 완료 여부와 무관하게 게시글은 "마감(시간 만료)"으로 자동 전환
- 방장은 언제든 게시글을 직접 취소할 수 있음 (참여자에게 알림)
- 참여자는 방장 승인 전까지는 신청을 자유롭게 취소 가능. 승인 후 취소 시 누적 금액에서 제외되고 방장에게 알림

### 4-5. 매칭 완료 후 참여자 제외 및 재모집
1. 매칭 완료 이후에도 방장은 채팅방/참여자 목록에서 특정 참여자를 강제로 제외할 수 있음 (무응답·잠적 대응)
2. 제외된 참여자의 예상 금액은 누적 금액에서 즉시 차감됨
3. 차감 결과 누적 금액이 최소 주문 금액 미만이 되면 게시글 상태는 자동으로 "모집중"으로 되돌아가고, 게시판에 다시 노출되어 신규 신청을 받을 수 있음
4. 반대로 차감 후에도 누적 금액이 최소 주문 금액 이상이면 "매칭 완료" 상태를 유지

## 5. 기능 요구사항 (MVP)

### 5-1. 인증/계정
- 학교 이메일 인증 기반 회원가입/로그인
- 최소 프로필: 닉네임, 인증된 학교 이메일 (실명/전화번호 등은 MVP에서 수집하지 않음)

### 5-2. 모임 게시글
- 작성 필드: 위치(건물 드롭다운 + 상세 텍스트), 식사 가능 시간대(시작~종료), 가게명, 메뉴 설명(자유 텍스트), 배달료, 최소 주문 금액, 방장 본인 메뉴 가격(예상 주문 금액), 비고(선택)
  - 배달료는 참여자들이 신청 여부를 판단할 때 참고하는 정보성 필드로, 최소 주문 금액 도달 계산(누적 예상 주문 금액 합산)에는 포함되지 않음
- 상태: `모집중` / `매칭완료` / `마감(시간만료)` / `취소됨` (매칭완료는 참여자 제외로 인해 다시 `모집중`으로 되돌아갈 수 있음, §4-5 참고)
- 게시글 상세: 가게/메뉴/배달료 정보, 현재 누적 금액 / 최소 주문 금액 진행률, 참여자 목록(승인된 인원만 공개, 닉네임 수준)

### 5-3. 게시판 목록/탐색
- 목록 정렬: 최신순(기본), 식사 시간 임박순
- 필터: 위치(건물), 시간대, 모집 상태
- "모집중" 게시글 우선 노출, 매칭완료/마감 게시글은 목록에서 제외 또는 별도 탭

### 5-4. 참여 신청 & 승인
- 참여자: 예상 주문 금액 입력 후 신청
- 방장: 신청 목록에서 승인/거절, 승인 시 누적 금액 자동 갱신
- 방장: 매칭완료 이후에도 참여자 목록에서 특정 참여자를 강제 제외 가능 (§4-5)
- 신청/승인/거절/매칭완료/제외 시점에 인앱 알림(최소한 배지 또는 알림함 형태)

### 5-5. 매칭 완료 로직
- 승인된 참여자 예상 금액의 합계가 최소 주문 금액 이상이면 자동으로 매칭완료 처리
- 매칭완료 전환 시, 그 시점까지 남아있던 대기(pending) 신청은 자동으로 거절 처리
- 매칭완료 후에는 신규 신청 불가. 단, 방장의 참여자 강제 제외로 누적 금액이 최소 주문 금액 미만이 되면 자동으로 "모집중"으로 재전환되어 신규 신청을 다시 받음 (§4-5)
- 방장이 최소 주문 금액 자체를 매칭완료 후에 수정하는 것은 MVP 범위 아님 (Out of scope)

### 5-6. 채팅
- 매칭완료된 모임 단위로 방장 + 승인된 참여자 전원이 포함된 **단일 그룹 채팅방** 자동 생성 (1:1 개별 채팅 아님)
- 외부 메신저로 이동하지 않고 웹사이트 내에서 실시간 송수신 (WebSocket 기반 실시간 통신)
- 텍스트 메시지 송수신 (이미지/파일 첨부는 MVP 범위 아님)
- 채팅방은 해당 모임 참여자만 접근 가능

### 5-7. 마이페이지
- 내가 작성한 모임 목록 (상태별)
- 내가 참여 중/참여했던 모임 목록
- 받은 참여 신청 관리 (방장 시점)

## 6. 비기능 요구사항

- **플랫폼**: 반응형 웹. 모바일 브라우저에서의 사용성을 우선 고려 (실사용은 대부분 모바일에서 발생할 것으로 예상)
- **성능**: 매칭 완료 판정은 참여 승인 즉시 실시간 반영되어야 함 (누적 금액이 최소 금액을 넘는 순간 바로 상태 전환)
- **보안/인증**: 학교 도메인 화이트리스트 기반 검증 후, 인증 링크/코드를 담은 메일을 자동 발송하는 방식으로 학교 이메일 인증 처리
- **알림**: MVP에서는 인앱 알림 수준으로 충분 (푸시 알림은 Out of scope)

## 7. 데이터 모델 초안 (개략)

```
User
 - id, nickname, school_email, school_id, created_at

Meetup (모임/게시글)
 - id, host_user_id, school_id
 - location_building, location_detail
 - meal_time_start, meal_time_end
 - store_name, menu_description
 - delivery_fee (정보성 필드, 매칭 완료 판정에는 미반영)
 - min_order_amount
 - status: recruiting | matched | expired | cancelled
 - created_at

Participation (참여 신청)
 - id, meetup_id, user_id
 - expected_amount
 - status: pending | approved | rejected | cancelled | removed
   (rejected: 방장 거절 또는 매칭완료 시 자동 거절 / removed: 매칭완료 후 방장의 강제 제외)
 - created_at

ChatRoom
 - id, meetup_id (1:1 매핑)

ChatMessage
 - id, chat_room_id, user_id, content, created_at
```

## 8. Out of Scope (v2 이후 고려)

| 항목 | 비고 |
|---|---|
| 인앱 정산(더치페이 계산) 기능 | 방향성은 확정(대표 1인 주문 후 정산 지원), MVP에서는 채팅으로 사용자가 자율 정산. v2에서 인원별 금액 계산 기능 추가 |
| 다중 대학 확장 | 데이터 모델은 `school_id`로 미리 대비하되, UI/가입 흐름은 단일 대학 기준으로 먼저 구축 |
| 노쇼/신뢰도(평점) 시스템 | 참여 후 잠적하는 사용자에 대한 패널티 정책은 추후 설계 |
| 배달앱 실시간 연동 | 배달의민족 등과의 메뉴/가격 연동은 하지 않음. 메뉴는 자유 텍스트로만 관리 |
| 매칭완료 후 최소 주문 금액 수정 | 방장이 임의로 최소 주문 금액을 올리거나 내리는 기능은 MVP 범위 아님 (참여자 강제 제외로 인한 자동 재모집만 지원, §4-5) |
| 푸시 알림 | 인앱 알림함으로 대체 |
| 신고/차단 기능 | 커뮤니티 안전 기능은 v2 검토 |

## 9. 결정 사항 (구현 전 확정)

| 항목 | 결정 |
|---|---|
| 식사 가능 시간 입력 형식 | 시간대 범위로 입력 (예: 12:00~12:30) |
| 매칭완료 시점의 대기 신청 처리 | 누적 금액이 최소 주문 금액을 넘어 매칭완료로 전환되는 즉시, 아직 승인되지 않은 대기(pending) 신청은 시스템이 자동으로 거절 처리 |
| 학교 이메일 인증 운영 (MVP) | 학교 도메인 화이트리스트를 두고, 가입 시 도메인 검증 후 인증 링크/코드가 담긴 메일을 자동 발송 |
| 매칭완료 후 이탈자 처리 | 방장이 채팅방/참여자 목록에서 특정 참여자를 강제 제외할 수 있음. 제외 시 해당 인원의 예상 금액은 누적 금액에서 차감되며, 그 결과 누적 금액이 다시 최소 주문 금액 미만이 되면 게시글은 "모집중" 상태로 되돌아가 신규/대기 신청을 다시 받을 수 있음 |

## 10. 화면 목록 (MVP)

1. 로그인/회원가입 (학교 이메일 인증)
2. 게시판 목록 (필터/정렬 포함)
3. 게시글 상세 (참여 신청, 진행률, 참여자 목록)
4. 모임 만들기
5. 참여 신청 관리 (방장용)
6. 그룹 채팅방
7. 마이페이지 (작성한 모임 / 참여한 모임)
